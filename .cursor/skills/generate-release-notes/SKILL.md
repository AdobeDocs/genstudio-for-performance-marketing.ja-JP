# GenStudio リリースノートの生成

**正規ターゲットファイル：** [help/user-guide/release-notes.md](help/user-guide/release-notes.md)

**完全な例：** [examples.md](examples.md)

**KT/wiki フィールドマッピングとドキュメントパス：** [reference.md](reference.md)

## 範囲の編集（厳格）

このスキルを使用する場合、**唯一の場所**&#x200B;あなたは&#x200B;**追加**&#x200B;または&#x200B;**編集** リリースノート本文コンテンツが&#x200B;**`## … {#latest}`** （`{#latest}` アンカーを運ぶ単一のブロック）で見出されるセクションです。

- **以前のリリースノート**&#x200B;を編集しない&#x200B;**– 任意の`+++Notes from YYYY.MM.DD+++`折りたたみ可能なブロック – または`{#latest}`を持たない**&#x200B;任意の&#x200B;**古い`##`月次セクション。トピックが関連していると思われる場合でも、リンクが間違っているように見える、またはコピーが重複または古くなっています。**
- **ユーザーがこのスキルでカバーされていない**&#x200B;明示的な&#x200B;**要求を行わない限り、現在の`{#latest}` ブロックの外にある`###` サブセクション、箇条書き、リンク、または文言を**&#x200B;前に「修正」しないでください。
- **例外：** [前の最新](#archive-previous-latest)をアーカイブして、**new**&#x200B;の上位`{#latest}` ブロックを導入します。**以前の`{#latest}` セクション全体を** new **の下に折り畳める**&#x200B;以前のリリースノート **に移動します。以下で説明します。**&#x200B;このパスの間、**は**&#x200B;その他の古い&#x200B;**のアーカイブ ブロックに書き換えまたは追加を**&#x200B;しません。

新しい情報がドキュメントに属する場合は、現在の&#x200B;**`{#latest}`**&#x200B;見出しの下に配置します（または最初にアーカイブしてから、新しい`{#latest}`の下にのみ追加します）。

## ワークフローチェックリスト

この順序で作業します。 チェックリストをコピーして、マルチステップ編集の進捗状況を追跡します。

1. [ ] `help/user-guide/release-notes.md`を開き、現在の`## YYYY.MM {#latest}` ブロックを読み取ります。 手順2でアーカイブ手順を実行しない限り、**以前のリリースノート**&#x200B;を&#x200B;**読み取り専用** コンテキストとして扱います。
2. [ ] **new**&#x200B;の月次リリースを追加する場合：現在の最新のリリースをアーカイブします（[以前の最新の](#archive-previous-latest)をアーカイブするを参照）。
3. [ ]上位`## YYYY.MM {#latest}` セクションの&#x200B;**のみ**&#x200B;を追加または編集します（最新の月はリリースリストの上部）。
4. [ ]各項目について、[決定ルール ](#decision-rules)を適用します（機能`###`対&#x200B;**修正と機能強化**、Beta バッジなど）。
5. [ ]最も関連性の高い語句のドキュメント リンクを追加または検証します（[reference.md](reference.md#documentation-linking)を参照）。
6. [ ]終了する前に[品質チェック ](#quality-checks)を実行します。
7. [ ] タスクがメタデータを明示的に更新しない限り、[frontmatter](reference.md#frontmatter)をページに保持します。

## 決定ルール

コンテンツを適切な場所に配置するために、次のif/then ルールを使用します。

| If | その後 |
|----|------|
| 機能は新しく、Betaでは | Beta バッジ行を`###`見出しの直下に追加します（[examples.md](examples.md)を参照）。 |
| Source マテリアルは、項目を&#x200B;**fix**&#x200B;または&#x200B;**enhancement**&#x200B;として明示的にラベル付けします | `*`個の箇条書きのみを使用して`### Fixes and enhancements`未満に置きます。 |
| アイテムは、まったく新しい機能または機能ストーリーです | 1 ～ 3個の文章を含む`###`機能セクションを使用します（修正リストは使用しません）。 |
| 何かが修正なのか機能なのかは不明です | ソースが修正/機能強化を明確に示さない限り、デフォルトは`###`機能セクションです。 |

**修正セクションルール：** ソースが修正または機能強化として明示的にラベル付けされていない限り、**修正および機能強化**&#x200B;に箇条書きを追加しないでください。

## 過去の最新情報をアーカイブ

新しい`## YYYY.MM {#latest}`の導入時：

1. 前の`## YYYY.MM {#latest}` セクション全体を切り取ります（次の`##`または&#x200B;**以前のリリースノート**&#x200B;の前に、そのリリースのコンテンツの末尾から見出しを切り取ります）。
2. **new**&#x200B;折りたたみ可能なブロック内の&#x200B;**以前のリリースノート**&#x200B;に貼り付けます。
3. 古い見出しを`+++Notes from YYYY.MM.DD+++`に置き換えます（実際のリリース日を使用します。ファイル内の既存のメモと同じ形式にします）。
4. アーカイブされた見出しから`{#latest}`を削除します。新しい上部セクションは`{#latest}`を持つ唯一のセクションです。
5. **以前のリリースノート**&#x200B;内の時系列を維持します（ファイルが既に別の順序を使用しない限り、最新のアーカイブ済みブロックは先頭に向かって移動します – **既存のファイル**）と一致します。

このアーカイブの実行中に、**not**&#x200B;は&#x200B;**既存の** `+++Notes from …+++` ブロックの本文を編集します。新しくアーカイブされたブロックのみを挿入し、古いアーカイブをそのまま保持します。

## 必要な構造

### ページタイトルと概要

frontmatterの後、次を使用します。

```markdown
# GenStudio for Performance Marketing release notes

This release information details the latest updates to the GenStudio for Performance Marketing application.
```

ファイルが既に少し異なるイントロ文を使用している場合（例：「詳細」ではなく「provides」を使用している場合）、ページの残りの部分との一貫性を維持します。

### 最新のリリース見出し

- 最新のリリース ブロックの形式：`## YYYY.MM {#latest}`。
- ページ上の`{#latest}` アンカーは1つだけです。

### 機能セクション

- 主要な機能カテゴリには`###`を使用します。
- 1～3個の文章を時制で表現し、何が/なぜ表示されるかを明確にし、利用者の行動を役立つ場所に配置します。
- 製品名：`[!DNL Create]`、`[!DNL Content]`、`[!DNL Insights]`など
- UI：適切な場合は`[!UICONTROL Control Name]`
- 強調：UI領域/セクションの場合は`_italics_`、主要用語の場合は`**bold**`を控えめにします。

### Beta バッジ

次を正確に使用：

```markdown
[!BADGE Beta]{type=Informative tooltip="This feature is currently in Beta, so some functionality may be limited or subject to change."}
```

### ドキュメントへのリンク

- パターン：`[link text](/help/user-guide/section/page.md#anchor)`
- アンカーを優先する。ユーザーが気にするフレーズをリンクし、「ここをクリック」をクリックしないでください。

### 修正と機能強化

- 箇条書き行では`*`を使用します。
- ソース素材の「修正/強化」というラベルが明示的に付けられた項目のみ。
- 機能と同じリンクおよび用語の規則。

## 禁止されているコンテンツ

- 公開されたリリースノートに、Jira キー、内部イシュー番号、内部専用URL、または企業のWiki リンクを&#x200B;**not**&#x200B;に含めないでください。
- **not**&#x200B;は、ナレッジ転送ドキュメント、チケット、または内部ツールをプルーフとして挙げ、ユーザーに向けた結果のみを要約します。
- **not**&#x200B;は、複数のセクションで`{#latest}`を複製します。

## 品質チェック

タスクを完了する前：

- [ ] **スコープ：** `## … {#latest}` ブロックのみが追加または編集されました。**以前のリリースノート**&#x200B;および以前の月次セクションは、[以前の最新の](#archive-previous-latest)のカット/ペーストを&#x200B;**新しい** アーカイブブロックにアーカイブすることを除き、変更されませんでした。`{#latest}`
- [ ]すべての新規または変更された相対リンクは、可能な場合、`help/`の下の実際のパスに解決されます。
- [ ]個のBeta機能には、必要に応じてBeta バッジスニペットが含まれます。
- [ ]用語が既存のリリースノート （`[!DNL …]`、`[!UICONTROL …]`）と一致します。
- [ ]誤った内部ID （`[A-Z]+-\d+`）、Wiki URL、または「Jiraを参照」言語のドラフトをスキャンして、削除します。
- [ ] **修正と機能強化**&#x200B;には、明示的にラベル付けされた修正/機能強化のみが含まれます。
- [ ]新しい月が追加されたときに、以前の最新セクションが正しくアーカイブされました。

## コンテンツソース（概要）

ナレッジ転送ドキュメントまたはリリース Wikiから内部的に取得する場合は、[reference.md](reference.md#internal-sources-kt-and-release-wikis)で説明されているようにフィールドをマッピングします。 付属のページは、スタンドアロンユーザードキュメントとして読む必要があります。

## オプション：新しいサブセクションを磨く

`{#latest}`の下に新しい`###` コンテンツを追加した後、コピーエディター形式のパスに[ ポーランドのリリースノート ](../polish-release-notes/SKILL.md)を実行します（特典のトーン、段落&#x200B;**あたり** 2-3文章、手続き上のハウツーなし） **のみ**。以前のリリースノートまたは既存のテキストでは、明示的に要求されない限り&#x200B;**ではなく**。****
